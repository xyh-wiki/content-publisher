# 内容生成与多平台发布流程

## 1. 目标

平台采用“一份可信主稿，多个平台派生版本”的内容模型。运营人员不需要为每个平台重复编写文章，也不会把 Markdown 原文直接粘贴到只接受普通文本的社交平台。

## 2. 标准流程

1. 在“项目与内容”导入 HTTPS Git 仓库。
2. 等待仓库分析任务完成，确认项目名称、描述、技术栈、分支和源版本。
3. 配置语言、目标读者、语气、长度、SEO 必选关键词、禁用关键词和章节；第一项关键词作为主关键词，生成 SEO 优化草稿。
4. 在文章页查看 SEO 质量分并编辑主稿；重点检查标题、Meta Description、关键词前置、H2/H3、FAQ 和来源链接，每次保存创建不可变版本。
5. Admin 审核通过，锁定发布基线。
6. 选择发布方式：
   - 已配置 API 账号：提交后台发布任务。
   - 无 API 或选择人工发布：进入平台发布工作区。
7. 人工发布工作区自动生成平台标题、正文和标签，可继续编辑并复制。
8. 点击官方入口，在新窗口完成平台登录、排版和发布。
9. 将平台文章 URL 回填系统，保存发布内容快照、操作人和发布时间。
10. 在发布中心通过文章/渠道矩阵核对覆盖率，并在统一记录中查看 API 状态、人工外链和失败原因。

## 3. 平台内容适配矩阵

| 平台类型 | 平台 | 输出规则 |
|---|---|---|
| Markdown 长文 | DEV、WordPress、Discourse、GitHub Discussions、Reddit、Hashnode、Medium、Ghost、CSDN、掘金、博客园、SegmentFault、V2EX、开源中国 | 保留 Markdown 结构；按渠道附加原文链接；API 渠道再转换为平台所需 JSON 或 HTML |
| 普通文本 | 小红书、知乎 | 去除 Markdown 标记，保留段落；小红书标题限制 20 字并追加话题标签 |
| 短帖 | Twitter/X、Mastodon | 使用标题、摘要、原文链接和少量标签；按 280/500 Unicode 字符限制截断 |

关键词会按主关键词、次级词、长尾问题词和场景词组织，再经过规范化、去重和数量限制。平台派生内容不会覆盖文章主稿。

## 4. API 发布

- 渠道账号只能由 Admin 创建。
- 凭据使用 AES-GCM 加密保存，页面与 API 不回显明文。
- 发布前再次执行账号状态、文章审核状态和渠道端点安全检查。
- 平台内容适配与人工工作区使用同一适配器，避免预览内容和实际 API 请求不一致。
- 外部 API 结果不确定时不盲目自动重试，防止重复发布。

## 5. 跳转登录发布

人工流程不保存平台用户名、密码、Cookie 或验证码，也不执行浏览器自动化。系统只提供：

- 确定性的渠道内容适配；
- 标题、正文和完整内容复制；
- 官方创作或登录页面跳转；
- 发布结果 URL 的平台域名校验；
- 最终内容快照、操作人和发布时间留档。

发布链接必须为 HTTPS，且域名必须与所选平台匹配。例如小红书记录只接受 `xiaohongshu.com` 域名。

## 6. 页面入口

| 页面 | 地址 | 用途 |
|---|---|---|
| 工作台 | `/` | 指标、流程和最近内容 |
| 项目与内容 | `/projects` | 导入仓库、生成主稿、任务跟踪 |
| 发布中心 | `/publishing` | 文章/渠道矩阵、API/人工统一记录、失败状态和内容队列 |
| 渠道管理 | `/channels` | 创建 API 账号、查看人工渠道目录 |
| 文章详情 | `/articles/{articleId}` | 编辑、审核、选择发布渠道 |
| 人工发布工作区 | `/articles/{articleId}/manual/{channelType}` | 平台适配、复制、跳转和结果回填 |

## 7. 统一发布记录

底层继续保留 API `publications` 与人工 `manual_publications` 两类事实表。应用层将它们映射为同一只读模型，统一包含文章、渠道、发布方式、状态、内容格式、外链、有限错误信息和时间字段。

- API 记录保留 `PUBLISHING`、`PUBLISHED`、`FAILED` 状态及渠道账号展示名称。
- 人工记录统一映射为 `MANUAL / PUBLISHED`，保留操作人和平台外链。
- 列表严格按租户查询并按更新时间倒序合并。
- 页面和统一查询接口不返回加密凭据，也不返回人工发布的完整正文快照。
- 发布矩阵只展示每篇文章在每个渠道的最新记录，历史记录仍保留在统一时间线中。
