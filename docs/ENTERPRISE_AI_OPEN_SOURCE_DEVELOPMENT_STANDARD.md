# 企业级使用 AI 开发开源项目规范

## 1. 文档信息

- 文档名称：企业级使用 AI 开发开源项目规范
- 适用对象：产品、架构、开发、测试、安全、运维、法务、开源项目维护者及外部贡献者
- 适用范围：使用生成式 AI、代码补全模型、智能代理或自动化工具参与设计、编码、测试、文档、发布和维护的开源项目
- 规范级别：企业级强制基线
- 更新原则：安全要求、依赖策略、AI 模型、工具链或开源许可证发生变化时必须评审更新

本文使用以下规范词：

- **必须（MUST）**：不满足即不得合并、发布或对外分发。
- **应该（SHOULD）**：原则上必须满足；偏离时需要记录原因和批准人。
- **可以（MAY）**：根据项目情况选择实施。

## 2. 目标与原则

本规范用于保证 AI 提升研发效率的同时，不降低软件质量、安全性、知识产权合规性和工程可维护性。

项目必须遵循以下原则：

1. **人类最终负责**：AI 不能作为代码作者、评审人、审批人或责任主体。提交人和批准人必须对最终结果负责。
2. **最小必要使用**：只向 AI 提供完成任务所需的最少上下文，不上传秘密、个人信息或未经授权的代码。
3. **事实可验证**：AI 输出属于未经验证的建议，不得直接视为事实、设计结论或可发布代码。
4. **安全默认开启**：身份、权限、网络、依赖、数据和发布流程采用最小权限与默认拒绝策略。
5. **全过程可追溯**：需求、设计、AI 使用、代码变更、评审、测试、发布及事故处理必须保留证据。
6. **开源合规优先**：任何代码、模型输出、数据集、依赖和素材在进入仓库前必须确认来源与授权。
7. **质量门禁一致**：AI 生成代码与人工代码执行完全相同的评审、测试、安全扫描和发布标准。
8. **可回滚和可恢复**：所有生产变更必须支持安全回滚，关键数据变更必须有备份与恢复验证。

## 3. 角色与职责

| 角色 | 主要职责 |
|---|---|
| 项目负责人 | 确定目标、范围、风险等级、资源和最终交付责任 |
| 产品负责人 | 维护需求、验收标准、用户影响和业务优先级 |
| 架构负责人 | 审核技术方案、模块边界、数据模型和非功能要求 |
| 开发人员 | 验证 AI 输出，完成实现、测试、文档和变更声明 |
| 代码评审人 | 独立检查正确性、安全性、兼容性和可维护性 |
| 安全负责人 | 维护威胁模型、安全基线、漏洞处置和发布准入规则 |
| 开源治理负责人 | 审核许可证、NOTICE、第三方来源和社区治理规则 |
| 运维负责人 | 负责部署、监控、告警、备份、回滚和事故响应 |
| 发布负责人 | 核对版本、制品、变更日志、签名和发布批准记录 |

高风险变更不得由同一人同时完成开发、最终评审和生产发布。

## 4. AI 工具与模型准入

### 4.1 工具准入

企业使用的 AI 工具必须登记以下信息：

- 工具名称、供应商、版本和使用入口；
- 模型名称、版本或可识别的模型系列；
- 数据存储位置、保留期限和删除机制；
- 输入数据是否用于模型训练；
- 身份认证、权限控制、审计日志和加密能力；
- 服务条款、隐私政策、知识产权条款和可用性承诺；
- 支持的数据分类级别和禁止输入类型；
- 退出方案、数据导出方式和替代工具。

未经批准的公网 AI 服务不得处理企业内部源码、未公开漏洞、客户数据、访问凭据或商业秘密。

### 4.2 模型变更

模型、系统提示词、代理权限或关键工具版本变更时，必须重新执行：

1. 功能回归测试；
2. 安全与提示词注入测试；
3. 敏感信息泄漏测试；
4. 代码质量基准测试；
5. 成本、延迟和可用性评估；
6. 变更记录和回滚验证。

## 5. 数据分类与输入边界

### 5.1 数据等级

| 等级 | 示例 | AI 使用要求 |
|---|---|---|
| 公开 | 已发布开源代码、公开文档 | 可使用批准工具处理 |
| 内部 | 未发布设计、普通内部代码 | 仅限企业批准且不训练模型的受控服务 |
| 机密 | 商业秘密、未公开路线图、私有算法 | 默认禁止；例外必须经过安全与法务批准 |
| 严格机密 | 密钥、Token、密码、私钥、客户敏感数据 | 任何情况下不得输入 AI |

### 5.2 强制禁止输入

以下信息不得进入提示词、附件、代理上下文、日志或模型缓存：

- 密码、API Key、Token、Cookie、私钥和恢复码；
- 未脱敏的个人信息、客户数据和生产数据库内容；
- 未公开漏洞细节、攻击路径和生产安全配置；
- 未经授权的第三方源码、付费资料和受限制数据集；
- 企业机密合同、财务信息和商业谈判材料。

项目必须启用提交前秘密扫描，并在 CI 中再次扫描 Git 历史和构建产物。

## 6. 开源项目立项规范

立项时必须形成并评审以下内容：

- 项目目标、非目标和目标用户；
- 功能范围、非功能需求和验收标准；
- 风险等级、威胁模型和数据分类；
- 开源许可证选择及其兼容性说明；
- 仓库归属、维护者、审批人和安全联系人；
- 依赖策略、发布策略和支持周期；
- AI 使用场景、允许工具、禁止数据和审计要求；
- 退出、归档和维护者交接方案。

仓库公开前必须完成敏感信息扫描、历史清理、许可证审核和内部信息审查。

## 7. 开源许可证与知识产权

1. 仓库根目录必须包含有效的 `LICENSE` 文件。
2. 使用第三方代码、模型、字体、图片、数据集或文档时，必须记录来源、版本、许可证和修改情况。
3. 必须维护 `NOTICE` 或第三方组件清单，满足归属声明要求。
4. 禁止复制来源不明的 AI 输出、问答网站代码或无法确认许可证的代码片段。
5. AI 生成内容不得自动声明为“原创”或保证无知识产权风险。
6. 引入 GPL、AGPL、SSPL、Commons Clause 或其他强约束许可证前，必须进行法务和架构评审。
7. 外部贡献必须遵循项目的 CLA 或 DCO 策略。
8. 发现疑似侵权内容时必须停止分发，隔离相关版本并启动处置流程。

## 8. 仓库与分支管理

### 8.1 基础文件

企业级开源仓库应该至少包含：

```text
README.md
LICENSE
NOTICE
SECURITY.md
CONTRIBUTING.md
CODE_OF_CONDUCT.md
CHANGELOG.md
.gitignore
.editorconfig
.github/CODEOWNERS
.github/pull_request_template.md
docs/
src/
tests/
```

### 8.2 分支策略

- 默认分支必须启用保护，不允许直接推送。
- 所有变更必须通过 Pull Request 合并。
- 必须要求至少一名独立评审人；安全关键模块要求安全负责人或 CODEOWNER 批准。
- CI、测试、安全扫描和许可证检查必须通过后才能合并。
- 禁止强制推送到受保护分支。
- 禁止通过关闭测试、降低阈值或删除检查来绕过质量门禁。

### 8.3 提交规范

提交应保持单一目的，推荐使用 Conventional Commits：

```text
feat: 增加内容审核工作流
fix: 修复租户数据越权查询
docs: 更新部署和回滚说明
test: 补充权限边界集成测试
refactor: 拆分渠道发布适配器
security: 升级存在漏洞的依赖
```

提交信息不得包含密钥、客户名称、漏洞利用细节或敏感内部地址。

## 9. AI 辅助开发流程

### 9.1 标准流程

每个 AI 辅助任务必须按以下顺序执行：

1. 明确需求、范围、验收标准和禁止事项；
2. 读取项目规范、架构说明和相关测试；
3. 将任务拆分为可独立验证的小变更；
4. 向 AI 提供最少且脱敏的上下文；
5. 检查 AI 建议是否符合现有架构和依赖策略；
6. 实施变更并补充测试、文档和迁移说明；
7. 执行格式、编译、单元测试、集成测试和安全检查；
8. 人工逐行评审 AI 生成代码；
9. 在 PR 中声明 AI 参与范围、验证方式和剩余风险；
10. 独立评审通过后合并。

### 9.2 提示词规范

提示词必须包含：

- 任务目标和明确边界；
- 允许修改的模块和文件；
- 现有架构、语言、框架和版本；
- 安全、性能、兼容性和数据要求；
- 验收标准及必须运行的测试；
- 禁止虚构 API、依赖、配置和测试结果；
- 不得覆盖无关用户修改；
- 输出中不得包含秘密和个人信息。

禁止使用“忽略所有限制”“直接修改生产环境”“跳过测试”等无约束提示词。

### 9.3 AI 代理权限

- 默认只授予读取权限。
- 写文件权限必须限制在明确的项目目录。
- 网络访问、依赖安装、数据库写入和生产操作必须单独授权。
- AI 不得自行合并 PR、创建正式发布、修改分支保护或轮换生产密钥。
- 删除、覆盖、数据迁移等破坏性操作必须先解析精确目标并保留备份。
- AI 运行日志不得记录密钥、完整生产数据或敏感提示词。

## 10. 架构与设计规范

1. 重要功能必须先形成可评审的设计说明或 ADR。
2. 模块必须职责单一，核心业务逻辑不得依赖具体 Web、数据库或第三方 SDK。
3. 外部系统必须通过清晰的端口或适配器隔离。
4. 数据模型、权限模型和状态机必须显式定义。
5. 接口必须定义输入、输出、错误码、幂等性、超时和重试策略。
6. 跨服务调用必须设置超时，不允许无限等待。
7. 重试必须区分可重试与不可重试错误，并使用退避和抖动。
8. 写操作必须考虑并发、幂等、事务和重复执行。
9. 配置与代码分离，环境差异不得通过复制代码实现。
10. 关键技术决策必须记录选型理由、替代方案、风险和回滚方式。

AI 提出的新框架、新依赖或大规模重构不得直接实施，必须经过架构评审。

## 11. 编码规范

### 11.1 通用要求

- 代码必须清晰、可读、可测试，避免过度抽象和无必要的设计模式。
- 公共接口、复杂算法、安全边界和非显然决策必须有说明。
- 禁止空异常处理、静默失败和仅记录不处置的严重错误。
- 错误信息不得泄漏密钥、SQL、内部路径、堆栈或用户隐私。
- 时间统一使用明确时区；金额使用定点类型；标识符不得使用浮点数。
- 输入必须在信任边界处校验，输出必须根据使用场景编码。
- 日志必须结构化并包含必要的请求、任务或追踪标识。
- 不得在业务代码中硬编码秘密、生产地址或租户配置。

### 11.2 AI 生成代码检查

开发人员必须重点检查：

- 是否调用了不存在、过期或不兼容的 API；
- 是否引入虚构配置项、错误参数或危险默认值；
- 是否破坏租户、角色、资源归属或数据权限边界；
- 是否存在 SQL 注入、命令注入、路径遍历、SSRF、XSS 或反序列化风险；
- 是否错误处理并发、事务、时区、编码和边界值；
- 是否吞掉异常、泄漏信息或产生不可观测失败；
- 是否复制重复逻辑或偏离现有架构；
- 测试是否只验证实现细节而未验证业务结果；
- 注释和文档是否与实际代码一致。

## 12. 安全规范

### 12.1 身份与权限

- 使用成熟认证协议，不自研密码学和身份协议。
- 所有服务端授权必须基于可信身份重新判断，不能信任前端隐藏按钮。
- 采用最小权限、默认拒绝和资源归属校验。
- 管理操作、密钥操作和高风险发布必须记录审计事件。
- 服务账号必须独立、可撤销、可轮换，不得多人共享。

### 12.2 应用安全

- 所有外部输入必须验证长度、格式、类型和允许范围。
- URL 访问必须防止 SSRF，并限制协议、主机、端口和重定向。
- 文件处理必须限制类型、大小、名称、解压比例和存储位置。
- 数据库必须使用参数化查询。
- HTML、Markdown 和富文本必须根据输出场景进行安全处理。
- 上传和构建产物必须执行恶意内容扫描。
- 安全响应头、CSRF、防暴力破解和速率限制必须按场景启用。

### 12.3 密钥管理

- 密钥存放在批准的密钥管理系统或受控环境文件中。
- 密钥不得进入源码、镜像层、日志、测试快照和错误响应。
- 必须定义轮换周期、吊销流程和泄漏应急流程。
- 发现泄漏时必须先吊销，再调查影响，不能只删除 Git 中的字符串。

## 13. 软件供应链与依赖

1. 新增依赖必须说明用途、许可证、维护状态、版本和替代方案。
2. 禁止使用来源不明、长期无人维护或存在未处置高危漏洞的组件。
3. 必须锁定可复现版本，避免无上限动态版本。
4. CI 必须执行依赖漏洞、许可证和恶意包检查。
5. 发布制品必须生成 SBOM，推荐使用 SPDX 或 CycloneDX。
6. 构建过程应使用固定基础镜像和最小化运行时镜像。
7. 发布制品、容器镜像和关键标签应该签名并保留来源证明。
8. 依赖升级必须通过回归测试；安全升级不得以“AI 判断无影响”代替验证。

## 14. 测试与质量门禁

### 14.1 必测范围

- 单元测试：核心规则、边界值和失败分支；
- 集成测试：数据库、消息、文件、身份和第三方适配；
- API 契约测试：字段、状态码、错误码和兼容性；
- 权限测试：跨用户、跨租户、跨角色和资源归属；
- 安全测试：注入、SSRF、XSS、路径遍历、秘密泄漏和滥用；
- 性能测试：关键接口延迟、吞吐、资源使用和容量边界；
- 恢复测试：重试、任务恢复、备份恢复、回滚和部分失败；
- 升级测试：数据库迁移、配置兼容和旧版本客户端。

### 14.2 AI 功能专项测试

使用 AI 的功能还必须测试：

- 提示词注入和间接提示词注入；
- 敏感信息回显和越权上下文访问；
- 幻觉、虚构引用和不可验证结论；
- 超长输入输出、编码异常和结构化输出破坏；
- 模型超时、限流、不可用和返回格式变化；
- 成本上限、并发上限和滥用防护；
- 模型版本变化导致的质量回归；
- 人工接管、降级和关闭 AI 后的业务可用性。

### 14.3 合并门禁

PR 合并前至少满足：

- 编译和静态检查通过；
- 必需测试全部通过；
- 高危和严重漏洞为零；
- 无新增秘密泄漏；
- 许可证检查通过；
- 关键路径有测试覆盖；
- 文档、迁移和变更日志已更新；
- 独立评审人批准；
- AI 参与情况已声明。

不得仅依据 AI 生成的测试证明 AI 生成的实现正确。

## 15. Pull Request 与评审规范

PR 描述必须包含：

```markdown
## 变更目的

## 实现说明

## AI 使用声明
- 使用工具与模型：
- AI 参与范围：
- 人工验证内容：
- 未采用的 AI 建议：

## 安全与数据影响

## 测试证据

## 兼容性与迁移

## 回滚方案

## 剩余风险
```

评审人不得只查看 AI 摘要，必须检查实际差异、测试和依赖变化。

以下变更需要额外评审：

- 身份、权限、密码学和密钥管理；
- 数据库迁移和数据删除；
- 网络访问、文件系统和命令执行；
- 依赖、构建、发布和 CI 权限；
- 日志、审计和个人信息处理；
- 开源许可证和第三方代码来源；
- AI 模型、提示词、工具权限和数据边界。

## 16. 文档规范

项目必须维护：

- README：定位、能力、限制、快速开始和支持范围；
- 架构文档：模块、数据流、信任边界和关键决策；
- API 文档：认证、请求、响应、错误和示例；
- 部署文档：环境、配置、依赖、升级、备份和回滚；
- 安全文档：漏洞报告渠道、支持版本和响应承诺；
- 贡献文档：环境搭建、分支、测试、提交和评审要求；
- 变更日志：新增、修复、安全、弃用和不兼容变化；
- AI 使用说明：工具范围、数据限制、人工责任和已知风险。

AI 生成文档必须由对应模块负责人核对，不得包含不存在的接口、参数、版本或命令。

## 17. CI/CD 与发布规范

### 17.1 CI

CI 应按最小权限运行，并执行：

1. 格式与静态分析；
2. 编译和测试；
3. 秘密扫描；
4. SAST 与依赖漏洞扫描；
5. 许可证检查；
6. SBOM 生成；
7. 构建制品和完整性校验；
8. 必要的集成与安全测试。

来自 Fork 的 PR 不得获得生产密钥或高权限发布令牌。

### 17.2 发布

- 使用语义化版本管理。
- 正式版本必须由受保护标签或批准工作流生成。
- 发布说明必须列出功能、修复、安全影响、升级步骤和不兼容变化。
- 制品必须可追溯到 Commit、构建任务、依赖清单和审批记录。
- 数据库迁移必须提供前置检查、备份、执行、验证和回滚说明。
- 发布前必须验证配置、容量、告警、备份和回滚能力。
- 禁止由 AI 在无人确认情况下自动发布生产版本。

## 18. 部署、可观测性与运维

- 服务必须提供健康、就绪和必要的依赖状态检查。
- 日志、指标和追踪必须使用统一关联标识。
- 关键指标必须覆盖流量、错误、延迟、资源、任务积压和外部依赖。
- 告警必须包含影响范围、处理入口和运行手册链接。
- 日志不得包含密钥、完整提示词、个人信息和敏感正文。
- AI 调用应记录模型标识、请求耗时、状态、Token 或成本摘要，但不得记录敏感输入。
- 必须设置调用超时、并发限制、速率限制、预算和熔断策略。
- 备份必须加密、定期验证恢复，并明确恢复点和恢复时间目标。

## 19. 漏洞与事故响应

项目必须提供私密漏洞报告渠道，并定义：

1. 接收与确认时限；
2. 风险分级；
3. 影响版本和资产识别；
4. 临时缓解措施；
5. 修复、测试和发布流程；
6. CVE 或安全公告策略；
7. 用户通知与升级建议；
8. 根因分析和防复发行动。

发生 AI 相关事故时，还必须确认：

- 是否存在敏感数据输入、模型缓存或日志残留；
- 是否由提示词注入、代理越权或工具调用导致；
- 是否需要停用模型、撤销令牌或关闭功能；
- 是否影响其他租户、仓库、版本或生成内容；
- 是否需要更新模型评测集和安全门禁。

## 20. 社区与外部贡献治理

- 必须发布贡献指南和行为准则。
- Issue 和 PR 模板不得诱导用户提交秘密或个人信息。
- 机器人和 AI 自动回复必须明确其自动化身份。
- AI 生成的外部贡献必须遵守相同质量与许可证要求。
- 维护者必须防范恶意依赖、提示词注入、构建脚本和 CI 工作流变更。
- 新维护者权限应逐步授予，发布和安全权限必须最小化。
- 长期无人维护时应明确归档、转交或停止支持状态。

## 21. 审计与度量

项目应保留以下证据：

- 需求、设计和风险评审记录；
- AI 工具、模型和使用范围；
- PR、评审、测试和安全扫描结果；
- 许可证、依赖和 SBOM；
- 发布批准、制品签名和部署记录；
- 漏洞、事故、恢复和改进行动。

推荐跟踪以下指标：

- 变更交付周期和失败率；
- 缺陷逃逸率和回滚率；
- 高危漏洞修复时间；
- 依赖更新及时率；
- AI 生成代码占比和人工修改比例；
- AI 建议采纳率与回退率；
- 提示词注入、安全和质量回归测试通过率；
- 文档完整率和版本一致性。

指标用于改进流程，不得用于鼓励未经验证地增加 AI 生成代码数量。

## 22. 禁止行为

以下行为属于严重违规：

- 将秘密、客户数据或未授权源码输入公网 AI；
- 未阅读代码即合并 AI 生成变更；
- 使用 AI 伪造测试结果、评审记录、引用、许可证或安全结论；
- 绕过分支保护、质量门禁或安全扫描；
- 允许 AI 自主访问生产密钥、执行生产删除或发布正式版本；
- 引入来源和许可证不明的代码；
- 隐瞒 AI 参与的高风险变更；
- 用关闭检查、删除测试或降低阈值掩盖失败；
- 将 AI 作为事故责任主体或最终审批人。

## 23. Definition of Done

功能只有在以下条件全部满足时才视为完成：

- [ ] 需求和验收标准已满足；
- [ ] 架构和安全边界未被破坏；
- [ ] AI 参与范围已声明并完成人工验证；
- [ ] 代码符合项目规范且无无关改动；
- [ ] 单元、集成、权限和失败路径测试通过；
- [ ] 静态分析、秘密、漏洞和许可证扫描通过；
- [ ] 文档、API、配置、迁移和变更日志已更新；
- [ ] 兼容性、性能和数据影响已评估；
- [ ] 回滚方案已验证；
- [ ] 独立评审和必要的专项审批已完成；
- [ ] 构建制品、SBOM、签名和来源可追溯；
- [ ] 不存在未说明的高风险遗留问题。

## 24. 发布前检查清单

### 24.1 代码与质量

- [ ] 默认分支保护有效；
- [ ] 所有必需检查通过；
- [ ] 高危与严重漏洞为零；
- [ ] 测试覆盖关键业务和安全边界；
- [ ] 无调试代码、测试密钥和临时开关；
- [ ] AI 生成内容已逐行复核。

### 24.2 开源与供应链

- [ ] LICENSE、NOTICE 和第三方清单完整；
- [ ] 新增依赖来源、版本和许可证已审核；
- [ ] SBOM 已生成；
- [ ] 制品和容器已签名；
- [ ] 仓库历史与制品不存在敏感信息。

### 24.3 部署与运维

- [ ] 配置和密钥通过受控方式提供；
- [ ] 数据迁移已备份并验证；
- [ ] 健康检查、指标、日志和告警有效；
- [ ] 容量与超时配置符合预期；
- [ ] 回滚和恢复步骤已演练；
- [ ] 值班人员和事故联系人明确。

## 25. 规范维护

本规范至少每六个月评审一次。出现以下情况时必须立即评审：

- 引入新的 AI 模型、代理或代码执行能力；
- 开始处理新的敏感数据类型；
- 开源许可证或供应链策略变化；
- 发生严重漏洞、数据泄漏或 AI 越权事故；
- 生产架构、发布流程或监管要求发生重大变化。

所有例外必须记录适用范围、原因、风险、补偿措施、批准人和失效日期。永久例外必须转化为正式规范变更。
